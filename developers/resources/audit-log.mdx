---
title: Audit Log
description: Reference for Discord audit log objects and endpoints for viewing administrative actions in a guild.
---

When an administrative action is performed in a guild, an entry is added to its audit log. Viewing audit logs requires the `VIEW_AUDIT_LOG` permission. All audit log entries are stored for 45 days.

When an app performs an eligible action, it can pass an `X-Audit-Log-Reason` header to indicate why the action was taken.

## Audit log object

### Audit log structure

<ResponseField name="application_commands" type="array of application command objects" required>
  List of application commands referenced in the audit log.
</ResponseField>

<ResponseField name="audit_log_entries" type="array of audit log entry objects" required>
  List of audit log entries, sorted from most to least recent.
</ResponseField>

<ResponseField name="auto_moderation_rules" type="array of auto moderation rule objects" required>
  List of auto moderation rules referenced in the audit log.
</ResponseField>

<ResponseField name="guild_scheduled_events" type="array of guild scheduled event objects" required>
  List of guild scheduled events referenced in the audit log.
</ResponseField>

<ResponseField name="integrations" type="array of partial integration objects" required>
  List of partial integration objects.
</ResponseField>

<ResponseField name="threads" type="array of channel objects" required>
  List of threads referenced in the audit log. Threads referenced in `THREAD_CREATE` and `THREAD_UPDATE` events are included since archived threads might not be kept in memory by clients.
</ResponseField>

<ResponseField name="users" type="array of user objects" required>
  List of users referenced in the audit log.
</ResponseField>

<ResponseField name="webhooks" type="array of webhook objects" required>
  List of webhooks referenced in the audit log.
</ResponseField>

## Audit log entry object

Each audit log entry represents a single administrative action indicated by `action_type`. Most entries contain changes in the `changes` array that affected an entity in Discord.

Apps can specify why an administrative action is being taken by passing an `X-Audit-Log-Reason` request header. The header supports 1–512 URL-encoded UTF-8 characters. Reasons are visible to users in the client and to apps when fetching audit log entries.

### Audit log entry structure

<ResponseField name="target_id" type="?string" required>
  ID of the affected entity (webhook, user, role, etc.).
</ResponseField>

<ResponseField name="changes" type="array of audit log change objects">
  Changes made to the `target_id`. Optional field.
</ResponseField>

<ResponseField name="user_id" type="?snowflake" required>
  User or app that made the changes.
</ResponseField>

<ResponseField name="id" type="snowflake" required>
  ID of the entry.
</ResponseField>

<ResponseField name="action_type" type="integer" required>
  Type of action that occurred. See [audit log events](#audit-log-events).
</ResponseField>

<ResponseField name="options" type="optional audit entry info object">
  Additional info for certain event types. Optional field.
</ResponseField>

<ResponseField name="reason" type="string">
  Reason for the change (1–512 characters). Optional field.
</ResponseField>

<Warning>
For `APPLICATION_COMMAND_PERMISSION_UPDATE` events, the `target_id` is the command ID or the app ID, since the `changes` array represents the entire `permissions` property on the guild permissions object.
</Warning>

## Audit log change object

Many audit log events include a `changes` array in their entry object. The structure of individual changes varies based on the event type.

<Info>
If `new_value` is not present while `old_value` is, the property has been reset or set to `null`. If `old_value` is not included, the property was previously `null`.
</Info>

### Audit log change structure

<ResponseField name="new_value" type="mixed">
  New value of the key. Type matches the object field's type. Optional field.
</ResponseField>

<ResponseField name="old_value" type="mixed">
  Old value of the key. Type matches the object field's type. Optional field.
</ResponseField>

<ResponseField name="key" type="string" required>
  Name of the changed entity, with a few [exceptions](#audit-log-change-exceptions).
</ResponseField>

### Audit log change exceptions

For most objects, the change keys correspond to the changed object's fields. The following table details exceptions.

| Object changed         | Change key exceptions                                                  | Change object exceptions                                                                                                                                                                               |
|------------------------|------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Command Permission     | Snowflake as key                                                       | The `changes` array contains objects with a `key` field representing the entity whose command was affected (role, channel, or user ID), a previous permissions object (`old_value`), and an updated permissions object (`new_value`) |
| Invite and Invite Metadata | Additional `channel_id` key (instead of `channel.id`)             |                                                                                                                                                                                                        |
| Partial Role           | `$add` and `$remove` as keys                                           | `new_value` is an array of objects containing the role `id` and `name`                                                                                                                                |
| Webhook                | `avatar_hash` key (instead of `avatar`)                                |                                                                                                                                                                                                        |

## Get guild audit log

<Info>
`GET /guilds/{guild.id}/audit-logs`
</Info>

Returns an [audit log](#audit-log-object) object for the guild. Requires the `VIEW_AUDIT_LOG` permission.

The returned list of audit log entries is ordered based on whether you use `before` or `after`:

- **`before`**: List is ordered by entry ID descending (newer entries first).
- **`after`**: List is reversed and appears in ascending order (older entries first).
- **Neither**: Defaults to `before` the current timestamp, showing the most recent entries in descending order. Use `after=0` to show the oldest entries instead.

### Query string params

<ParamField query="user_id" type="snowflake">
  Filter entries to a specific user ID.
</ParamField>

<ParamField query="action_type" type="integer">
  Filter entries for a specific [audit log event](#audit-log-events).
</ParamField>

<ParamField query="before" type="snowflake">
  Entries with ID less than a specific audit log entry ID.
</ParamField>

<ParamField query="after" type="snowflake">
  Entries with ID greater than a specific audit log entry ID.
</ParamField>

<ParamField query="limit" type="integer" default="50">
  Maximum number of entries to return (1–100).
</ParamField>
